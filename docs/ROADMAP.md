# ROADMAP — API 연동 카탈로그

> [`docs/PRD.md`](PRD.md)에서 정의한 "무엇을 만들지"를, **어떤 순서로** 만들지 정리한 문서.
> 핵심 원칙: **골격 → 공통 → 개별 기능**. 기능부터 만들지 않는다.

## 전체 순서 한눈에 보기
```
Phase 1 골격      프로젝트 셋업 + Notion 연동 환경 + 기본 레이아웃
   ↓
Phase 2 공통      데이터 레이어(lib/notion, types) + 재사용 컴포넌트(카드/뱃지/검색·필터)
   ↓
Phase 3 핵심 기능  API 목록 페이지 + 상세 페이지(Notion 블록 렌더링)
   ↓
Phase 4 추가 기능  검색 + 카테고리/메서드/상태 필터 + 카테고리별 페이지
   ↓
Phase 5 최적화/배포 ISR + 반응형/다크모드/접근성 + Vercel 배포
```

> 💡 이 프로젝트의 순서 결정에서 가장 중요한 지점: **Phase 2의 데이터 레이어(`lib/notion.ts` + `types/api.ts`)가 모든 화면의 기반**이다. 목록·상세·필터가 전부 같은 조회 함수와 타입을 쓰므로, 이걸 먼저 만들지 않으면 화면마다 Notion 파싱 코드가 중복된다.

---

## Phase 1 — 프로젝트 초기 설정 (골격) · 약 0.5일

**작업**
- Next.js 16 (App Router) + TypeScript 프로젝트 생성
- Tailwind CSS + shadcn/ui + Lucide React 설치/초기화
- `@notionhq/client` 설치, `.env.local`에 `NOTION_API_KEY`·`NOTION_DATABASE_ID` 자리 구성
- 기본 레이아웃(`app/layout.tsx`): 상단 네비 + 다크모드 토글 + 푸터, 글로벌 스타일

**왜 이 순서인가**
견고한 구조와 환경이 없으면 이후 모든 작업이 흔들린다. 환경 변수·레이아웃 같은 골격을 먼저 고정한다.

**완료 기준**
- `npm run dev`로 빈 레이아웃이 뜨고 다크모드 토글이 동작
- 환경 변수 로딩 구조가 잡혀 있음(키는 아직 더미 가능)

---

## Phase 2 — 공통 모듈/컴포넌트 (공통) · 약 1~1.5일

**작업**
- **데이터 레이어** `lib/notion.ts`: Notion 클라이언트 초기화, `fetchApis()`(목록), `fetchApiById(id)`(상세 + 블록), 에러/빈값 처리
- **타입 매핑** `types/api.ts`: Notion 속성(Name/Category/Method/Endpoint/Provider/Auth/Status/Tags/Version/Updated) → 앱 타입 `ApiItem`으로 변환하는 단일 매퍼
- **공통 UI**: `ApiCard`, 메서드/상태용 `Badge`, `SearchInput`, `FilterSelect`, `Header`/`Footer`
- 공통 유틸(메서드별 색상, 상태 라벨 등)

**왜 이 순서인가**
목록·상세·필터가 **모두 같은 조회 함수·타입·뱃지**를 재사용한다. 공통을 먼저 만들어야 개별 기능에서 중복이 안 생기고, Notion 스키마 변경 시 한 곳만 고치면 된다. (PRD 리스크 "스키마 변경으로 매핑 깨짐" 대응)

**완료 기준**
- `fetchApis()`가 실제 Notion DB에서 항목을 가져와 `ApiItem[]`로 반환
- `ApiCard`에 더미/실데이터를 넣으면 카드가 정상 렌더링

---

## Phase 3 — 핵심 기능 (개별 기능) · 약 1~1.5일

**작업**
- **목록 페이지** `app/page.tsx`: `fetchApis()` 결과를 `ApiCard` 그리드로 표시
- **상세 페이지** `app/apis/[id]/page.tsx`: 헤더에 메서드·엔드포인트·상태 뱃지, 본문은 Notion 블록(문단·코드블록·표) → React 렌더링

**왜 이 순서인가**
Phase 2의 공통 기반 위에 올리는 "가장 기본이 되는 가치". 목록과 상세가 있어야 카탈로그가 최소한으로 작동한다.

**완료 기준**
- 목록에서 카드를 누르면 해당 API 상세로 이동
- 상세에서 엔드포인트·인증·파라미터·요청/응답 예시를 확인 가능

---

## Phase 4 — 추가 기능 (개별 기능) · 약 1일

**작업**
- 검색: 이름·엔드포인트·태그 키워드 필터
- 필터: 카테고리 / HTTP 메서드 / 상태(운영·베타·지원종료)
- 카테고리별 페이지 `app/category/[name]/page.tsx`

**왜 이 순서인가**
핵심(목록·상세)이 완성된 뒤 탐색성을 높이는 부가 기능. 검색·필터는 Phase 2의 `SearchInput`/`FilterSelect`와 동일한 `ApiItem` 데이터를 그대로 활용한다.

**완료 기준**
- 키워드 검색과 3종 필터가 동시에 동작
- 카테고리 페이지가 해당 카테고리 항목만 표시

---

## Phase 5 — 최적화 및 배포 · 약 0.5~1일

**작업**
- ISR `revalidate` 설정으로 Notion 변경 자동 반영
- 반응형/다크모드 마감, 접근성(키보드 조작·텍스트 뱃지 병행)
- Notion API rate limit 대비 캐싱 점검
- Vercel 배포 + 환경 변수 설정

**왜 이 순서인가**
기능이 완성된 후 품질·성능·배포를 다룬다. 완성 전 최적화는 낭비다.

**완료 기준**
- 배포된 URL 접속 가능, Notion에 항목 추가 시 재검증 주기 내 반영
- 모바일에서 레이아웃이 깨지지 않음

## 단계별 의존 관계 요약
| Phase | 선행 조건 | 산출물 |
|---|---|---|
| 1 골격 | - | 실행되는 빈 앱 + 레이아웃 |
| 2 공통 | 1 | 데이터 레이어 + 타입 + 공통 컴포넌트 |
| 3 핵심 | 2 | 목록/상세 페이지 |
| 4 추가 | 2, 3 | 검색/필터/카테고리 |
| 5 배포 | 1~4 | 배포된 카탈로그 |

총 예상: **약 4.5~6일** (1인 기준, MVP)
