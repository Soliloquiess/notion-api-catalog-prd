# PRD — API 연동 카탈로그 (Notion CMS 기반)

> 사내/팀의 외부·내부 API 연동 명세를 **Notion에서 관리**하고, 웹에서 **검색·필터 가능한 카탈로그**로 보여주는 서비스.
> 비개발자(기획·운영)도 Notion에서 API 정보를 추가/수정하면 사이트에 자동 반영된다.

---

## 1. 프로젝트 개요

| 항목 | 내용 |
|---|---|
| 프로젝트명 | API 연동 카탈로그 (API Catalog Hub) |
| 목적 | 흩어진 API 연동 명세를 한곳에 모아 **검색·탐색 가능한 카탈로그**로 제공 |
| CMS 선택 이유 | API 명세는 자주 바뀌고 여러 사람이 수정한다. Notion을 CMS로 쓰면 **비개발자도 콘텐츠 관리**가 가능하고, 별도 DB·관리화면 구축 없이 시작할 수 있다. |
| 대상 사용자 | 사내 개발자(API 사용자), 연동 담당자, 기획/운영자 |

### 왜 이 주제인가 (배경)
핀테크/연동이 많은 환경에서는 "이 기능 쓰려면 어떤 API를, 어떤 인증으로, 어떤 파라미터로 호출하지?"를 매번 찾는 비용이 크다.
명세를 Notion DB로 관리하고 웹에서 검색·필터로 빠르게 찾도록 하면 이 반복 비용을 줄인다.

---

## 2. 주요 기능

1. **API 목록 조회** — Notion 데이터베이스의 API 항목을 카드/리스트로 표시
2. **검색** — 이름·엔드포인트·태그 기준 키워드 검색
3. **필터** — 카테고리 / HTTP 메서드 / 상태(운영·베타·지원종료) 별 필터링
4. **상세 페이지** — 개별 API의 엔드포인트, 인증 방식, 파라미터, 요청/응답 예시(Notion 본문) 렌더링
5. **자동 반영** — Notion 수정 시 ISR로 일정 주기마다 사이트에 반영
6. **반응형 + 다크모드** — 데스크톱/모바일, 라이트/다크 지원

---

## 3. 기술 스택

| 영역 | 선택 | 비고 |
|---|---|---|
| Frontend | Next.js 16 (App Router) + TypeScript | 서버 컴포넌트로 Notion 데이터 페칭 |
| CMS | Notion API (`@notionhq/client`) | API 키는 **서버에서만** 사용 |
| Styling | Tailwind CSS + shadcn/ui | 카드·뱃지·입력 컴포넌트 재사용 |
| Icons | Lucide React | 메서드/카테고리 아이콘 |
| 데이터 갱신 | ISR (Incremental Static Regeneration) | `revalidate`로 주기적 재생성 |
| 배포 | Vercel | 환경 변수로 Notion 키 주입 |

> 🔒 보안 포인트: Notion API 키와 데이터베이스 ID는 **서버 측 환경 변수**(`NOTION_API_KEY`, `NOTION_DATABASE_ID`)로만 다룬다. 클라이언트 번들에 절대 포함하지 않는다. (SPA가 아닌 Next.js 서버 렌더링을 택한 핵심 이유)

---

## 4. Notion 데이터베이스 구조

데이터베이스 1개(`APIs`)로 시작한다.

| 필드 | Notion 타입 | 설명 |
|---|---|---|
| Name | title | API/엔드포인트 이름 (예: 계좌 잔액 조회) |
| Category | select | 도메인 분류 (예: 인증 / 계좌 / 결제 / 마이데이터 / 스크래핑) |
| Method | select | HTTP 메서드 (GET / POST / PUT / DELETE) |
| Endpoint | rich_text | 요청 경로 (예: `/v1/accounts/{id}/balance`) |
| Provider | select | 제공처/연동 기관 |
| Auth | select | 인증 방식 (API Key / OAuth2 / mTLS 등) |
| Status | select | 상태 (운영 / 베타 / 지원종료) |
| Tags | multi_select | 자유 태그 (검색용) |
| Version | rich_text | API 버전 (예: v1) |
| Updated | last_edited_time | 마지막 수정 시각 (자동) |
| (본문) | page content | 상세 설명, 파라미터 표, 요청/응답 예시 코드블록 |

## 5. 화면 구성

| 화면 | 경로 | 설명 |
|---|---|---|
| 홈/목록 | `/` | API 카드 그리드 + 검색창 + 카테고리/메서드/상태 필터 |
| 상세 | `/apis/[id]` | 개별 API 명세 (헤더에 메서드·엔드포인트·상태 뱃지, 본문 블록 렌더링) |
| 카테고리 | `/category/[name]` | 특정 카테고리의 API 목록 |
| (공통) | 레이아웃 | 상단 네비 + 다크모드 토글 + 푸터 |

### 핵심 화면 와이어(텍스트)
```
[ 검색창...................... ]   [카테고리▼][메서드▼][상태▼]
┌───────────────┐ ┌───────────────┐ ┌───────────────┐
│ GET  계좌 잔액 │ │ POST 결제 요청 │ │ GET  거래내역  │
│ /v1/.../balance│ │ /v1/payments  │ │ /v1/.../txns  │
│ [운영] [계좌]  │ │ [베타] [결제]  │ │ [운영] [계좌]  │
└───────────────┘ └───────────────┘ └───────────────┘
```

## 6. MVP 범위

**포함 (Must)**
- Notion API 연동 + 환경 변수 설정
- API 목록 페이지 (카드 표시)
- 검색 + 카테고리/메서드/상태 필터
- API 상세 페이지 (본문 블록 렌더링)
- 반응형 + 다크모드

**제외 (Out of Scope, 추후)**
- 로그인/권한 관리 (사내 SSO 연동)
- Notion 외 직접 편집(웹에서 쓰기)
- API 호출 테스트(try-it-out) 콘솔
- 다국어, 즐겨찾기, 변경 이력 추적

---

## 7. 비기능 요구사항
- **성능**: 목록은 ISR로 정적 제공, 상세는 ID별 정적 생성 + `revalidate`
- **보안**: Notion 키 서버 전용, 클라이언트 노출 금지
- **접근성**: 검색/필터 키보드 조작 가능, 색만으로 정보 전달 금지(텍스트 뱃지 병행)
- **유지보수**: Notion 스키마 ↔ 앱 타입(`types/api.ts`)을 한 곳에서 매핑

---

## 8. 구현 단계

1. **프로젝트 셋업** — Next.js 16 + TS, Tailwind, shadcn/ui, Lucide, `@notionhq/client` 설치 / `.env.local` 구성
2. **Notion 준비** — `APIs` 데이터베이스 생성, 통합(Integration) 만들고 DB 공유, `NOTION_API_KEY`·`NOTION_DATABASE_ID` 발급
3. **데이터 레이어** — `lib/notion.ts`(목록·상세 조회) + `types/api.ts`(Notion 속성 → 앱 타입 매핑)
4. **목록 페이지** — 카드 컴포넌트 + 검색/필터 (서버에서 데이터, 클라이언트에서 필터링)
5. **상세 페이지** — `/apis/[id]`, Notion 블록 → React 렌더링(코드블록·표·문단)
6. **마감** — 다크모드/반응형 정리, ISR `revalidate` 설정, Vercel 배포

---

## 9. 성공 기준 (MVP 완료 정의)
- Notion에 항목을 추가하면 재검증 주기 내에 사이트 목록에 나타난다
- 키워드 검색과 3종 필터가 동작한다
- 상세 페이지에서 엔드포인트·인증·파라미터·예시를 확인할 수 있다
- 모바일에서 레이아웃이 깨지지 않고 다크모드가 동작한다

## 10. 리스크 & 대응
| 리스크 | 대응 |
|---|---|
| Notion API rate limit | ISR 캐싱으로 호출 최소화, 빌드시 일괄 조회 |
| Notion 스키마 변경으로 매핑 깨짐 | `types/api.ts`에 매핑 집중, 누락 필드 기본값 처리 |
| API 키 노출 | 서버 전용 사용, `.env.local`는 `.gitignore` 처리 |
